# 核心架构与差异化逻辑层设计

> 本文回答一个问题：这个系统和"通用 Agent + 一堆运维工具"的本质区别是什么。
> 答案由四个相互咬合的设计构成：编译时智能、环境事实库、安全执行层、验证飞轮。

## 0. 总览

```
渠道层        CLI / Web / 钉钉 / 飞书 / 企微 / Slack
                │
Gateway       会话、审批流、审计日志、多租户
                │
Agent 运行时   运行时模型只做"选择题"（见 §2）
                │
    ┌───────────┼───────────────┬─────────────┐
Skill 引擎   安全执行层        环境事实库     确定性工具层
（决策树）  （分级/简报/回滚） （自动CMDB）  （解析器/差分/规则）
                │
连接器层      SSH / NETCONF / kubectl / Prometheus / SNMP / IPMI / 云API
```

## 1. 编译时智能 vs 运行时选择（核心命题）

系统的聪明程度取决于**资产质量**，而不是运行时那个模型的智商。

| | 编译期（离线，不计成本） | 运行期（在线，可能是 7B 本地模型） |
|---|---|---|
| 谁 | Fable 5 / Opus 4.8 / 人类专家 | 用户配置的任意模型（含 Ollama 本地） |
| 做什么 | 编写 Skill、设计决策树、对抗评审、仿真验证 | 走树、选分支、判读、讲人话 |
| 产出 | 验证过的 Skill、解析器、语法树 | 一次排障/变更会话 |

推论：
- **Skill 格式按最弱运行时模型设计**（黄金准则 R7）。
- 运行时 80% 的步骤是机械的，由便宜模型执行；只有走出决策树边界才升级（§4）。
- 核心资产（Skill 库、解析器、事实模型）是**模型无关**的，基础模型换代不折旧。

## 2. 运行时的三档交互模型（信任阶梯）

| 档位 | Agent 做什么 | 人做什么 | 所需凭据 |
|---|---|---|---|
| 导航（Navigator） | 出方案、变更简报、判读人贴回的输出 | 执行一切命令 | 无 |
| 协驾（Copilot） | 执行只读命令与 dry-run | 执行写操作 | 只读 |
| 自驾（Autopilot） | 执行写操作 | 审批高危动作 | 读写 |

设计约束：
- 每个 Skill 必须三档皆可运行；档位是**会话属性**，不是 Skill 属性。
- 导航档的输出（步骤方案 + 简报）是可分享的文档——这是产品的自然传播机制。
- 升档需要显式配置，绝不因"这次比较急"临时提升。

## 3. 环境事实库（Environment Fact Store）

只读发现的产出不是一次性上下文，而是持续维护的结构化事实库（本质是自动化 CMDB）：

- **事实（fact）**：`os.family=linux`、`bgp.peers[]`、`fs.mounts[]`、`gpu.driver.version`……
  带采集时间戳与来源命令。
- Skill 在 `requirements.facts` 中声明依赖的事实；会话开始时按需采集或从缓存读取。
- 平时后台巡采，战时零探索——这同时解决了"指导要贴合环境"（R4）和"故障时刻延迟敏感"两个问题。
- 事实库全部本地存储，永不上传（air-gapped 友好是本项目对商业产品的决定性优势）。

## 4. 模型分层路由与升级飞轮

```
用户问题 → 匹配 Skill？
  ├─ 是 → 便宜模型走决策树 → 结论 / 或走到 otherwise
  └─ 否（或 otherwise）→ 升级：
        ├─ 用户配置了强模型 API → 强模型自由推理（协驾档以下）
        └─ 否则 → 明确告知"超出已验证知识范围"，转人工
```

**每一次升级事件 = 一个新 Skill 的候选。**
升级会话的 trace 自动归档为 skill-draft 原料；编译流水线（强模型）把它整理成决策树 →
仿真验证 → 社区评审 → 入库。系统越老越便宜、越老越不依赖强模型。

配套的反馈闭环：每次会话结束只问一个问题（"解决了吗 👍/👎"），
结果写入该 Skill 的 attestation 记录（见 `05-certification.md`）。

## 5. 确定性工具层

模型不裸眼读原始输出（R9）。三类确定性组件：

1. **解析器库**：网络设备 CLI → 结构化（复用 ntc-templates，补国产设备 VRP/Comware）；
   系统命令输出 → JSON（df、ss、ip、nvidia-smi……）。
2. **命令语法树**：按平台维护合法命令前缀树。任何将呈现给用户或将被执行的命令，
   先过语法校验——这是对"CLI 幻觉"的工程化拦截（R10）。
3. **规则与差分引擎**：阈值判断、基线对比、配置 diff、计数器增量判读。

## 6. 安全执行层

- **能力分级**：`read < low_risk_write < high_risk_write < destructive`，由连接器静态声明，
  Skill 的 action 节点标注 `risk`，两者在执行时取交集校验。
- **变更简报**：`risk >= medium` 的动作执行前生成（R6），作为审批卡片推送到 IM。
- **自动回滚**：`verify` 失败或简报中 `abort_if` 条件触发 → 执行 `rollback`（见 `02`）。
- **全量审计**：每条命令、每个模型决策、每次审批落审计日志，可回放。

## 7. 与现有生态的关系（不重造轮子）

- 执行面复用：Ansible / Netmiko / kubectl / nornir。
- K8s 诊断可集成 K8sGPT / HolmesGPT 作为连接器，而非重写。
- 我们的增量价值 = Skill 资产 + 安全执行模型 + 认证体系 + 事实库，不在底层工具。

## 8. 差异化总结（一句话版本）

> 别人是"把大模型接到运维工具上"；我们是"把运维专家的判断编译成可验证、可回滚、
> 可认证的资产，让任何模型——包括跑在你内网的小模型——都能安全地使用它"。

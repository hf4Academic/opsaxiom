# 可回滚保证机制（Rollback Design）

> 落实黄金准则 R1。目标：让"每个写操作都可撤销"成为被机器强制的不变量，
> 而不是被提示词请求的美德。

## 1. 回滚类型学（Rollback Taxonomy）

每个 `action` 节点的 `rollback.type` 必须是以下四种之一：

| type | 含义 | 典型例子 | 要求 |
|---|---|---|---|
| `inverse` | 存在确定的逆操作 | `ip link set eth0 down` ↔ `up`；`gzip` ↔ `gunzip` | `rollback.run` 必填，且在仿真中执行过 |
| `snapshot` | 操作前取快照，回滚=恢复快照 | 配置文件备份、etcd 快照、LVM 快照、网络设备 config backup | `snapshot.run`（取快照）与 `rollback.run`（恢复）都必填 |
| `transaction` | 平台原生事务性变更 | JunOS `commit confirmed 10`、`kubectl rollout undo`、数据库事务 | 必须声明确认窗口 `window` 与确认命令 `confirm.run`；超时未确认由平台自动回退 |
| `service_restore` | 无状态逆操作，但可通过服务级手段恢复 | 重启进程（无法"取消重启"，但可验证服务恢复） | `verify` 必填且必须包含服务健康断言；`risk` 自动 ≥ medium |

**不存在 `type: none`。** 写不出回滚方案的操作不允许成为可执行的 action 节点。

## 2. 不可逆操作的改造模式（核心设计）

天然不可逆的操作必须先改造成可逆形态。标准改造模式：

| 不可逆操作 | 改造后 | 回滚 |
|---|---|---|
| `rm 大文件` | `mv` 到隔离区 `/var/lib/opsagent/quarantine/<会话ID>/`（保留原路径清单） | 按清单 `mv` 回原位 |
| 覆盖配置文件 | 先 `cp` 带时间戳备份，再写入 | 恢复备份 |
| 清空日志/计数器 | 先导出到隔离区 | 导入/参照 |
| 缩容/删除资源 | 先导出资源定义（`kubectl get -o yaml`） | `apply` 回来 |
| DROP TABLE 等 | **不改造** | `human_only: true`，Agent 只出指导 |

隔离区由 Agent 管理生命周期：默认保留 7 天后提示人工确认清理——**最终删除永远是人的动作**。

## 3. 全链路强制点

回滚保证在四个层面各设一道闸，任何一道失守都还有下一道：

### 3.1 编译期（schema 校验器，CI 强制）
- `action` 节点缺 `rollback` → 校验失败，不能入库。
- `rollback` 与 `run` 的平台键必须一致（linux 的操作不能配 cisco 的回滚）。
- `risk >= medium` 而缺 `preflight` → 校验失败。

### 3.2 验证期（仿真环境，晋级 sim_verified 的门槛）
- 测试用例必须包含 **回滚路径断言**：执行 action → 执行 rollback → 断言系统状态等价于执行前。
- 回滚没测过 = 回滚不存在。

### 3.3 执行期（运行时引擎）
- 执行任何 action 前，先物化回滚方案（快照实际已取、逆命令实际已渲染并过语法树校验），
  物化失败则**拒绝执行原操作**。
- 变更简报中必须包含渲染后的回滚命令原文，人审批时看到的就是真的会执行的。
- `verify` 失败或 `abort_if` 触发 → 自动执行回滚 → 报告回滚结果（回滚本身也要 verify）。
- `transaction` 型：确认窗口内 Agent 主动监控 `preflight.watch` 指标，全部健康才发送确认命令。

### 3.4 会话期（导航档同样适用）
- 导航档下人来执行，但简报和回滚命令同样先行给出；
  Agent 在指导每个写操作之前，先指导人完成快照/备份步骤，顺序由决策树结构保证。

## 4. 回滚的边界与诚实声明

必须向用户明确的三件事（写入产品文案与每份简报模板）：

1. **回滚恢复的是"我们改动的部分"**，不是时间倒流。操作期间系统产生的新状态（新连接、新写入）不在回滚范围内。
2. `service_restore` 型操作存在服务中断窗口，简报中必须给出预估时长。
3. 分布式场景下回滚可能部分成功——回滚执行报告必须逐目标列出成功/失败，失败目标立即升级人工。

## 5. 与认证体系的联动

- attestation（实地验证记录）中单独设 `rollback_exercised: true/false` 字段——
  回滚在真实环境中被实际触发并成功的 Skill，认证权重显著更高。
- 任何一次真实环境的回滚失败报告 = 立即降级该 Skill 至 `draft` 并冻结自驾档执行，直到复审。
